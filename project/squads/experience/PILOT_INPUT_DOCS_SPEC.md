# Pilot Input-Documentation Spec

The pilot runs a real RFP cycle interactively (a Claude skill driving an MCP server over the governed
Postgres store), in parallel with the manual process. At every step it **generates the document the
sponsor fills out**, ingests it back **by key** (D20/D21), and after every engine run produces a
**clearly-versioned alignment file** (history kept in the schema). Post-award, negotiation
adjustments are **versioned** with an explicit version heading (ADR-0014).

The guiding loop: **generate fill-out doc → sponsor fills → ingest (key-validated) → act → produce
versioned output → keep history.** Predictable in process; every artifact reproducible and audited.

## Step 0 — Cycle Setup / Kickoff workbook  (pilot GENERATES blank → sponsor fills → ingest creates the cycle)

One workbook, one tab per dimension, header row carries field names + an inline note; the pilot
writes a blank, validated template (dropdowns where the domain is closed) and ingests the filled
copy into the governed `ref/cyc/perf` tables, creating the cycle + scope.

| Tab | Sponsor fills | Lands in |
|---|---|---|
| **Cycle** | commodity, sub-commodity, cycle label, horizon (weeks), # rounds, target effective date, weight preset (or custom weights), max suppliers/DC, premium ceiling, concentration threshold, coverage floor | `cyc.cycle` + the run `EngineConfig` |
| **DCs** | DC name, region (East/South/West…), state | `ref.dc` |
| **Lots / Items** | lot name, item description, pack size/UOM, **product type (Conventional/Organic)**, category | `ref.item` + `cyc.cycle_lot` + `cyc.cycle_lot_item` |
| **Suppliers (invited)** | supplier name, region/origin, notes | `ref.supplier` + `cyc.cycle_invited_supplier` |
| **Volumes** | DC × Lot × TF: weekly cases, weeks (→ period cases) | `cyc.cycle_projected_volume` |
| **Incumbents** | DC × Lot: incumbent supplier, **routing baseline $/case (prior-period actual-paid)**, contract notes | `perf.historical_award_assignment` + the routing baseline |
| **Timeframes** | TF label (season), start/end, week count | `cyc.cycle_timeframe` |
| **Premiums** *(optional)* | per-lot premium threshold override + rationale | `cyc.*` lot threshold |
| **Scenario rules** *(optional)* | Exclusions / Preferred / VolumeLimits (for lenses E/F/G) | `cyc.*` scenario-rule tables |
| **Safeties** *(optional, ADR-0014)* | per cell/lot: collar cap/floor, rolling-midpoint cadence, tolerance band, disaster triggers | `cyc.cycle_safety` |

Keys are generated by the pilot on ingest and embedded so later docs cascade by key (D21); the
sponsor only ever types **names** (D23). On ingest the pilot reports what it created and quarantines
any unresolved rows (never silently drops).

## Step 1 — Bid template per round  (pilot GENERATES owned template, keys embedded → sponsor/suppliers fill → key-validated ingest)

For each round the pilot generates the **owned bid template** for the cycle scope with **key IDs
embedded at line level** (D21) — the same format the engine ingests (D20). One row per
(DC × Lot × TF × invited supplier). Sponsor (or each supplier) fills: All-In $/case OR the
component split (FOB + Delivery + VegCool − Lot Discount), weekly + total volume offered, comments,
invested-flag (R1). The pilot ingests by key, validates, writes `bid.bid_line`, quarantines
MISSING_KEY/UNKNOWN_KEY rows, and reports counts. No-bid = blank price (recorded, not dropped).

## Step 2 — Run + the VERSIONED alignment file  (after every run)

The pilot runs the engine on the round (reads by key → `EngineInputs` → `V3Engine.run` → seals
`eng.analysis_run` + `bid_score` + `analysis_scenario` + `analysis_scenario_award`, incl. the
authoritative `rec_type`, D28), then generates the **alignment workbook** — the 18-tab scenario tool
— with a **clear version heading**: `MID-CYCLE ALIGNMENT ANALYSIS — {cycle} · Round {n} · Analysis
v{seq} · sealed {timestamp}`. Each run is a sealed, immutable historical version in
`eng.analysis_run`; v{seq} is the run's ordinal for the cycle. The sponsor can re-run a round (new
version) and compare versions; nothing is overwritten.

## Step 3 — Optional override docs (for lenses E/F/G)  (generate → fill → ingest → re-run)

If the sponsor wants to steer a scenario: the pilot generates a small Exclusions / Custom-award /
Preferred-supplier doc (names, with keys embedded), ingests it, and re-runs (a new alignment
version). This is how E (exclusions), F (custom overrides), G (preferred) are exercised on real data.

## Step 4 — Award selection → FREEZE  (sponsor picks a scenario; pilot freezes it)

The sponsor selects a scenario (B by default, or a custom build). The pilot **freezes** it into
`awd.award` + `awd.award_line` (the immutable baseline) and generates the **booking guide** (D22,
two audiences). Decision-support only — the human asserts the award (ADR-0006); the engine only
recommended.

## Step 5 — Post-award negotiation adjustments — VERSIONED  (freeze-and-layer, ADR-0014)

Each post-award negotiation/reprice is recorded as a new **version** layered on the frozen award
(`awd.award_adjustment` v1, v2, …; raw `award_line` never overwritten). The pilot generates an
**adjustment intake doc** (cells + new prices + type + effective date + reason), ingests it as the
next version, and produces the **post-award adjustments workbook** with an explicit heading:
`POST-AWARD ADJUSTMENTS — {award} · Version {N} · as of {date}`, a Versions history tab, the current
effective prices, and this version's changes. Version control is visible; every layer is auditable
and reproducible.

## Step 6 — History  (always in the schema)

Everything persists in the governed store: the cycle + scope, every round's bids, every sealed
analysis run (each a version), the frozen award, and every post-award adjustment version. The pilot
exposes history queries (list cycles, a cycle's rounds + analysis versions, an award's adjustment
versions) so the sponsor can pull any historical version's records or regenerate its document.

## MCP tool surface (the skill orchestrates these)

`cycle_setup_template()` · `cycle_create_from_setup(file)` · `bid_template(cycle, round)` ·
`ingest_bids(cycle, round, files)` · `run_round(cycle, round, config?)` ·
`alignment_file(cycle, run?)` · `override_template(cycle, kind)` · `ingest_overrides(...)` ·
`select_award(cycle, run, scenario, code)` · `booking_guide(award)` ·
`adjustment_template(award)` · `record_adjustment(award, file)` ·
`post_award_file(award, version?)` · `list_cycles()` · `cycle_history(cycle)` ·
`award_versions(award)`. Each returns names not keys (D23), is deterministic, and writes/reads the
governed store. The skill walks the sponsor step by step, telling them which doc to fill next.
