## 1. The two-auth-system finding (read this first — it frames every route)

There are **TWO completely separate auth mechanisms** in the codebase, and only one is wired to the routes:

1. **`app.core.security.*` (Principal + RBAC + TenantContext)** — the *designed* security boundary. A
   `Principal` carrying `roles: frozenset[Role]`, an RBAC permission matrix (`ROLE_PERMISSIONS`), a
   `require_permission(Permission)` FastAPI dependency factory, and `get_principal`/`get_tenant`. The
   principal is meant to be set on `request.state.principal` by a token-verification edge adapter (DEP-4,
   not built). Today only a **dev-principal stub** in `backend/app/main.py:34-47,77-78` sets it (in
   non-production), and production sets it to `None` (`main.py:75`).

2. **`app.auth.*` (cookie session → `CurrentUser`)** — the *actually-wired* console auth. `CurrentUser =
   Annotated[AppUser, Depends(get_current_user)]` (`backend/app/auth/deps.py:60`) reads the signed
   `kr_session` JWT cookie, loads `auth.app_user`, and 401s on any failure. **This is what every real route
   in this slice depends on.**

### PROOF that core/security RBAC is defined-but-UNENFORCED (drift)

`grep -rn 'Depends(require_permission|Depends(get_principal)|Depends(get_tenant)|require_permission('` over
`backend/app/api/**` → **NO MATCHES.** Not one route attaches a permission guard.

- `backend/app/api/deps.py:10-12` *re-exports* `get_principal, get_tenant, require_permission` from
  `app.core.security.deps` (so routers *could* import them from one place) — but **no router imports those
  three names.** They are exported and never consumed by the HTTP surface.
- Every authenticated route in `runs.py`, `bids.py`, `auth.py` takes `user: CurrentUser` — i.e. it proves
  *a valid session exists* but performs **zero permission/role check**. A signed-in user with any (or no)
  role can hit `POST /runs/{slug}/awards/freeze` (designed `AWARD_FREEZE`, Approver-only) exactly as easily
  as a list endpoint. The author≠approver separation of duties encoded in `rbac.py:62-119` is **not enforced
  anywhere in the request path.**
- The only consumers of `app.core.security` types **anywhere** in `backend/app` are: `main.py` (dev-principal
  stub) and `backend/app/domain/ref/service.py:16,25` (a constructor that *accepts* `TenantContext` +
  `Principal` but is a domain object, out of B5 scope). **No route, no middleware, enforces a `Permission`.**

This is the single largest decision-vs-code gap in the slice and is called out again per-file under §6
(rbac.py) and §7 (the runs routes). It maps to **ADR-0009 / security-PLAN §2** (the RBAC matrix) being
**modelled but not yet wired** to the live cookie-session console — consistent with the codebase comments
("Until the IdP adapter lands (DEP-4)…").

- **Content:** single module docstring `"""The HTTP surface (contract-first, /api/v1)."""`.
- **What:** package marker for `app.api`.
- **WHY:** Python needs `__init__.py` to treat `api/` as an importable package; the docstring states the

  it:** `from app.api.router import api_router` (used by `main.py`) would fail to resolve the package. Not

### 2.2 `backend/app/api/deps.py` (census #47, 486 B)

- **Inputs→outputs:** pure re-export module; no logic.
- **GOTCHA / drift:** it exports `get_principal`/`get_tenant`/`require_permission`, but (see §1) **only

- **Deps:** `app.core.db.session`, `app.core.security.deps`. Census cross-ref: #47.

## 3. `app/api/v1` — the live routers

Every route in this section authenticates via `user: CurrentUser` (cookie session) UNLESS noted
(health is unauthenticated; auth/login + auth/logout are pre-session). **No route attaches an RBAC
permission guard** (§1).

### 3.1 `backend/app/api/v1/health.py` (census #55, 965 B)
- **What:** two **unauthenticated** observability probes.
- **WHY:** PLAN §7 observability. Liveness vs readiness are split deliberately: liveness must NOT touch the
  DB (so a DB outage doesn't get the process killed), readiness MUST touch the DB (the store is the product,
  so "ready to serve" gates on DB reachability). Both unauthenticated so an orchestrator (k8s/Cloud Run) can
  probe without a session. **What breaks without it:** no liveness/readiness signal → orchestrator can't tell
  a hung process from a healthy one, and can't gate traffic on DB readiness.

| Method | Path | Auth | Request | Response | Status | Delegates to |
|---|---|---|---|---|---|---|
| GET | `/health` | none | — | `{"status":"ok"}` (`dict[str,str]`) | 200 | nothing — returns a literal |
| GET | `/ready` | none | — (injects `db: Session = Depends(get_db)`) | `{"status":"ready"}` | 200 on success; on DB failure the exception propagates → `_handle_unexpected` → **500 INTERNAL** problem envelope | `db.execute(text("SELECT 1"))` |

- **Edge cases:** `/ready` issues a literal `SELECT 1` (`health.py:32`). If the DB is unreachable, SQLAlchemy
  raises; there is no try/except, so it falls through to the catch-all handler (`handlers.py:83`) → a 500
  `internal_error` problem envelope (NOT a tailored "not ready" code). The docstring acknowledges this ("the
  error handlers surface a problem"). Census cross-ref: #55.

### 3.2 `backend/app/api/v1/auth.py` (census #50, 7428 B) — the console auth surface
- **What:** username/password sign-in (+ optional TOTP-2FA), logout, `/me`, and 2FA enrol/verify. The router
  is **request/response + cookie I/O ONLY** — all crypto/domain logic lives in `app/auth/security.py` (B6).
- **WHY:** the console needs a real session. httpOnly + Secure + SameSite cookie (`kr_session`) so the JWT is
  never exposed to JS (XSS-resistant) and rides cross-origin to the API. Two-step 2FA (enrol then verify)
  so a secret is never "on" until the user proves they can produce a code. **What breaks without it:** no way
  to authenticate the console; every `CurrentUser` route would 401.
- **Module constant:** `TWO_FACTOR_REQUIRED_DETAIL = "2FA code required"` (line 37) — a distinct 401 detail
  so the UI can tell "needs a 2FA code" apart from "bad credentials" and prompt for the code instead of
  failing the whole login.

**Request/response models (every field + validation):**
- `LoginRequest`: `username: str`, `password: str`, `totp_code: str | None = None` (no length/format
  constraints — TOTP validity is checked by `verify_totp`, not Pydantic).
- `UserView`: `id: str`, `username: str`, `totp_enabled: bool`. **Never** carries `password_hash` or
  `totp_secret` (docstring line 52 — the identity returned to the console excludes secrets).
- `LoginResponse`: `user: UserView`.
- `EnrollResponse`: `otpauth_uri: str`, `secret: str` (the provisioning material the UI renders as a QR +
  manual-entry secret).
- `VerifyRequest`: `code: str` (Field description "The 6-digit TOTP code"; no regex constraint).
- `VerifyResponse`: `totp_enabled: bool`.

**Helpers:** `_user_view(user)` → `UserView`; `_cookie_samesite()` reads `settings.auth_cookie_samesite`,
lowercases/strips it, narrows to `{"lax","strict","none"}` (default `"lax"` if invalid) — required because
`set_cookie`'s `samesite` arg only accepts those three literals; `_set_session_cookie(response, user)` mints a
token via `create_session_token(str(user.id), username=...)` and sets the `kr_session` cookie with
`max_age=session_cookie_max_age_seconds()`, `httponly=True`, `secure=settings.auth_cookie_secure`,
`samesite=_cookie_samesite()`, `path="/"`; `_unauthorized(detail)` → `AppError(UNAUTHENTICATED, …, 401)`.

| Method | Path | Auth | Request model | Response model | Status / errors | Delegates to |
|---|---|---|---|---|---|---|
| POST | `/auth/login` | **none** (pre-session) | `LoginRequest` | `LoginResponse` | 200; **401 UNAUTHENTICATED** on bad user/inactive/bad password (one opaque message "Invalid username or password." — no user enumeration); **401** with detail `"2FA code required"` when 2FA enabled and code missing/invalid | `verify_password`, `verify_totp` (app/auth/security); sets cookie |
| POST | `/auth/logout` | none | — | bare `Response` | **204 No Content**; idempotent (safe with no session) | `response.delete_cookie(...)` matching the set attributes (path, secure, samesite) so the browser actually clears it |
| GET | `/auth/me` | **`CurrentUser`** | — | `UserView` | 200; 401 if no valid session (raised inside `get_current_user`) | `_user_view(user)` |
| POST | `/auth/2fa/enroll` | `CurrentUser` | — | `EnrollResponse` | 200 | `generate_totp_secret()` → stores `user.totp_secret`, sets `user.totp_enabled=False`, `db.flush()`; returns `totp_provisioning_uri` + secret |
| POST | `/auth/2fa/verify` | `CurrentUser` | `VerifyRequest` | `VerifyResponse` | 200; **400 VALIDATION_ERROR** if no enrolment in progress (`user.totp_secret` falsy); **401** if code wrong | `verify_totp(user.totp_secret, body.code)` → sets `user.totp_enabled=True`, `db.flush()` |

- **Login branch detail (auth.py:128-141):** ONE lookup by username; `if user is None or not user.is_active
  or not verify_password(...)` → all three collapse to the SAME 401 (anti-enumeration). Only after the
  password passes does it branch on `user.totp_enabled`: missing code → 401 "2FA code required"; wrong code
  → 401 "2FA code required" (same detail, so the UI re-prompts rather than declaring failure).
- **DECISION points:** the distinct 2FA detail string is the UX-level "gate, not a failure" pattern; the
  opaque-401 is the security/PLAN anti-enumeration rule. Census cross-ref: #50.

### 3.3 `backend/app/api/v1/pilot_common.py` (census #57, 4386 B) — shared service wiring
- **What:** NOT a router (defines no routes). The shared `PilotService` wiring + run/round resolvers used by
  BOTH `runs.py` and `bids.py`, so neither router duplicates the wiring or reimplements domain logic.
- **WHY (critical to the whole API design):** ADR-0018 — the console drives the SAME `PilotService` the MCP
  harness wraps, with `isolate_db=False` so it shares the request's governed session (no per-run DB is
  provisioned). Factoring it here means run-identity resolution + the gate-vs-validation error discipline is
  written ONCE and reused. The test suite monkeypatches `_vault_root` on THIS module to redirect the vault.
  **What breaks without it:** runs/bids would each re-wire the service and could drift on the gate/validation
  error mapping (the exact thing the standard cares about).
- **Functions:**
  - `_vault_root() -> Path` (`@lru_cache`): `Path(settings.vault_root).expanduser()`, `mkdir(parents=True,
    exist_ok=True)`, returned. WHY cached + created-on-first-use: a fresh box "just works" without a manual
    mkdir; the cache means the dir is resolved once per process.
  - `service() -> PilotService`: `PilotService(_vault_root(), isolate_db=False, db_runs=True,
    persist_outputs=False)`. **The three flags are load-bearing** (per the docstring): `isolate_db=False`
    (share the request session, no per-run DB); `db_runs=True` (ADR-0018 Slice 2 — dual-write run identity to
    `pilot.run` so the stateless web path can resolve/list runs from the DB); `persist_outputs=False`
    (Slice 5 — governed ops do DB writes only and skip vault side-effects; every deliverable renders on
    request → honors CLAUDE.md req #4 "no server-side file storage"). The MCP harness builds its own service
    with different flags (no `db_runs`, `persist_outputs` default on, its own file vault).
  - `resolve_run(db, slug) -> Run`: `get_run(db, slug)`; **404 NOT_FOUND** (`AppError`, status 404) if no
    `pilot.run` row. WHY (ADR-0018 Slice 3): a run EXISTS iff it has a DB row — identity is DB-resolved, not
    folder-resolved.
  - `resolve_paths(db, slug) -> RunPaths`: calls `resolve_run` (404 if absent) then `service().run_paths(slug)`.
    Existence checked against the DB; the returned `RunPaths` still points at the vault location for the
    dual-write era.
  - `resolve_round_id(db, paths, round_no) -> str`: the central **gate-vs-validation discriminator.**
    - reads `run.cycle_id` from the `pilot.run` row; if empty → **400 GATE_REQUIRED** ("has no cycle yet —
      ingest the setup workbook first"). (`pilot_common.py:91-96`)
    - `load_cycle(db, cycle_id)`; if `round_no > len(cycle.rounds)` → **400 VALIDATION_ERROR** ("Round {n} is
      out of range — the cycle has {len} round(s)"). (`pilot_common.py:98-105`)
    - returns `cycle.rounds[round_no - 1].id` (1-based → 0-based index).
    - **WHY the two distinct codes:** an out-of-range round must never be mislabeled "no cycle yet" — they are
      different user actions (ingest setup vs pick a valid round). This is the AUDIT_STANDARD "gate_required
      vs validation_error" branch, enforced here once and reused by every round-scoped route.
- Census cross-ref: #57.

### 3.4 `backend/app/api/v1/runs.py` (census #58, 39661 B) — THE BIG ONE
- **What:** the runs surface (`/api/v1/runs`): list/create/read runs, list/download/zip a run's deliverables,

  list analyses, compare/inspect scenarios, freeze an award, finalize a run, list/inspect awards, record a
  post-award adjustment, and three comms-draft reads. **24 routes.** Wraps `PilotService` via
  `pilot_common.service()`; **no engine/domain logic is reimplemented** — every route is a thin
  request→service→view shim.
- **WHY:** this is the entire console-driven RFP loop's HTTP front end. Each route is authenticated
  (`CurrentUser`) and maps cleanly onto a `PilotService` method so the web path and the MCP path stay
  behaviorally identical (same service, same governed writes). **What breaks without it:** the console has no
  way to drive a run; the MCP harness would be the only way to operate the system.

  AWD-2026-TOMATO-1").

- `_run_archive_zip(db, run)` (lines 305-318): in-memory `zipfile.ZipFile(BytesIO, "w", ZIP_DEFLATED)`;

### 3.5 `backend/app/api/v1/bids.py` (census #52, 12966 B)
- **What:** the bids surface (`/api/v1/bids`): import a round's bids (strict or flexible) + list a round's

### 5.2 `backend/app/core/db/` — the persistence foundation

#### `db/__init__.py` (census #84, 186 B)
- Re-exports `Base, SchemaBase, metadata` from `db.base`; `__all__` accordingly. WHY: one import surface for
  the declarative base + the shared `MetaData`.

#### `db/base.py` (census #85, 2260 B) — declarative base + 8-schema convention
- **What:** the SQLAlchemy `DeclarativeBase` (`Base`) sharing ONE `MetaData`, plus `SchemaBase(schema)` — a
  factory returning an abstract per-layer base bound to a PostgreSQL schema.
- `SCHEMAS = ("ref","norm","cyc","bid","eng","awd","perf","audit")` — the **eight logical layers = eight
  real PG schemas** (PLAN §1.7/§2). Every mapped class lives in exactly one.
- `NAMING_CONVENTION` (lines 22-28): deterministic constraint/index names so Alembic autogenerate is
  reproducible and round-trips cleanly (kills SQLite-isms / autogenerate drift):
  - `ix`=`ix_%(table_name)s_%(column_0_N_name)s`, `uq`=`uq_…`, `ck`=`ck_%(table_name)s_%(constraint_name)s`,
    `fk`=`fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s`, `pk`=`pk_%(table_name)s`.
  - `metadata = MetaData(naming_convention=NAMING_CONVENTION)`; `Base.metadata = metadata`.
- `SchemaBase(schema) -> Any`: raises `ValueError("unknown layer schema …")` if `schema not in SCHEMAS`;
  else builds `class _LayerBase(Base): __abstract__ = True; __table_args__ = {"schema": schema}`, renames it
  `<Schema>Base`, returns it. Returns `Any` because the base is built at runtime (mypy can't treat it as a
  static base). **WHY:** domain packages do `class Foo(RefBase): …` and inherit the schema without repeating
  `__table_args__`. **What breaks without it:** every model would have to hand-write `{"schema": …}` and the
  naming convention; constraint names would drift across migrations.
- Census cross-ref: #85.

#### `db/session.py` (census #86, 1800 B) — **engine + the unit of work**
- **What:** the SQLAlchemy `Engine`, the `sessionmaker`, the request-scoped `unit_of_work()`, and the FastAPI
  `get_db()` dependency.
- `_build_engine()` → `create_engine(settings.database_url, echo=False, pool_pre_ping=True, future=True)`.
  `pool_pre_ping=True` validates a pooled connection before use (survives DB restarts). `engine` is built at
  import time (module-level singleton).
- `SessionLocal = sessionmaker(bind=engine, autoflush=False, autocommit=False, expire_on_commit=False,
  future=True)`. `autoflush=False` + `expire_on_commit=False` so objects stay usable after commit and flushes
  are explicit.
- **`unit_of_work()` (the transaction boundary, lines 43-59):** a `@contextmanager` that `yield`s a session,
  `commit()`s on success, `rollback()`s on ANY exception, and ALWAYS `close()`s in `finally`. **This is the
  single commit owner.** Transaction discipline (PLAN §7): **services `add`+`flush`, NEVER `commit`.** The UoW
  owns the boundary so a multi-service op is atomic and the **audit write lands in the SAME transaction as
  the change it records** (the entire tamper-evidence guarantee depends on this — if the audit row committed
  separately, a crash between the change and the audit could leave them inconsistent). **What breaks without
  it:** services would each commit, breaking atomicity and the audit-in-same-txn invariant.
- `get_db()` (lines 62-66): FastAPI dependency — `with unit_of_work() as session: yield session`. Every route
  that injects `Depends(get_db)` rides exactly one UoW per request; the commit/rollback happens after the
  route returns/raises. Census cross-ref: #86.

#### `db/types.py` (census #87, 1831 B) — **shared column types incl. the money precision**
- **What:** canonical column building blocks reused across all eight layers.
- **`MONEY_PRECISION = 18`, `MONEY_SCALE = 6`, `Money = Numeric(18, 6, asdecimal=True)`** — **the slice's
  single most important Layer-1 precision fact.** Six decimal places carry **case-level pricing without float
  drift** (the docstring). Every money value in the system (FOB, all-in $/case, adjustments, volume minimums)
  is stored at `Numeric(18,6)`. This is the DB end of the decimal journey that `runs.py:914`
  (`Decimal(str(float))`) and `bids.py:_num` (`float(Decimal)`) bracket on the API side.
- `uuid_pk()` → `mapped_column(PG_UUID(as_uuid=True), primary_key=True, default=uuid.uuid4)` — server-/app-
  generatable UUID PK.
- `money_column(*, nullable=True)` → `mapped_column(Money, nullable=...)` — a `Numeric(18,6)` money column.
- `client_id_column()` → non-null tenant FK to `ref.client.id`, `ondelete="RESTRICT"`, `nullable=False`,
  `index=True`. WHY: tenant-scoped tables carry it; the value is stamped from request context, never a
  client body (security/PLAN §1); indexed because every scoped read filters on it. RESTRICT so a client can't
  be deleted out from under its data.
- `created_at_column()` → `DateTime(timezone=True)`, `nullable=False`, `server_default=func.now()` (UTC).
- **What breaks without it:** money would risk float storage (drift → reconciliation failures, CLAUDE.md
  req #3); tenant FKs + UUID PKs would be re-declared inconsistently. Census cross-ref: #87.

### 5.3 `backend/app/core/audit/` — the live hash-chain

#### `audit/__init__.py` (census #77, 292 B)
- Re-exports `DomainEvent, EventType` (events) + `AuditWriter, compute_event_hash` (writer). WHY: one import
  surface for the audit producer side.

#### `audit/events.py` (census #78, 1690 B) — domain event types
- **What:** the `EventType(StrEnum)` catalog of governed state changes + the `DomainEvent` frozen dataclass.
- `EventType`: CREATED, SEALED, FROZEN, SUPERSEDED, SIGNED_OFF, SENT, GATE_APPROVED, IMPORTED, **CLOSED**
  (the run's terminal close-out — finalize; comment ties it to the design's "Finalize & close" step). These
  are the governed mutations (security/PLAN §3).
- `DomainEvent` (frozen, slots): `event_type:EventType`, `client_id:uuid`, `entity_type:str`, `entity_id:uuid`,
  `actor:str`, `cycle_id:uuid|None=None`, `source:str="api"` (api|worker|import), `before:dict|None=None`,
  `after:dict|None=None`, `metadata:dict=field(default_factory=dict)`. **WHY:** services emit one of these on
  every governed mutation and the single `AuditWriter` subscriber turns it into a hash-chained row in the
  same txn — population is NOT the caller's job to remember. `before`/`after` carry the canonical state so
  the writer hashes the transition without re-reading the row. **No commercial values in `metadata`** (it may
  surface in reads — security/PLAN §5). Census cross-ref: #78.

#### `audit/writer.py` (census #81, 5905 B) — **THE hash-chained writer (internals)**
- **What:** the single audit writer: one `DomainEvent` → one hash-chained `audit.event_log` row, in the
  caller's txn. Wired once (security/PLAN §3), not sprinkled across services.
- `GENESIS_HASH = "0"*64` — the prev-hash link for a tenant's first event.
- `_canonical(payload)` (line 32-35): **deterministic JSON** — `json.dumps(payload, sort_keys=True,
  separators=(",",":"), default=str)`. Sorted keys + compact + str-coerce non-JSON types → the SAME bytes
  for the same logical state, which is what makes the hash reproducible/verifiable.
- `hash_state(state)` → `None` if state is None, else `sha256(_canonical(state).encode()).hexdigest()`.
- **`compute_event_hash(...)` (lines 46-79) — the chain link (pure, deterministic):**
  `body = _canonical({client_id, seq, event_type, entity_type, entity_id, actor, occurred_at.isoformat(),
  before_state_hash, after_state_hash})`; returns `sha256((body + prev_event_hash).encode()).hexdigest()`.
  So **`event_hash = sha256(canonical(this row's fields) || prev_event_hash)`** — any edit/reorder/delete
  breaks the recomputed link (tamper-evidence, security/PLAN §3). Pure → unit-testable without a DB.
- **`AuditWriter` class:**
  - `__init__(session)`.
  - `_current_head(client_id) -> (prev_hash, last_seq)` (lines 88-104): `SELECT event_hash, seq FROM
    audit.event_log WHERE client_id=:cid ORDER BY seq DESC LIMIT 1 **FOR UPDATE**`. The **`FOR UPDATE` row
    lock serializes the per-tenant chain** so concurrent writers cannot fork it. Returns `(GENESIS_HASH, 0)`
    when the tenant has no events. (Note: literal SQL with bound params — no f-string interpolation — to
    satisfy bandit/ruff S-rules.)
  - **`append(event, occurred_at=None) -> str` (lines 106-160):** `occurred_at = occurred_at or
    datetime.now()`; `prev_hash, last_seq = _current_head(event.client_id)`; `seq = last_seq + 1`;
    `before_hash = hash_state(event.before)`, `after_hash = hash_state(event.after)`; `event_hash =
    compute_event_hash(...)`; then a literal `INSERT INTO audit.event_log (id, client_id, occurred_at, actor,
    source, event_type, entity_type, entity_id, cycle_id, before_state_hash, after_state_hash, prev_event_hash,
    event_hash, seq) VALUES (...)` with `id=uuid4()`, `cycle_id = str(cycle_id) if cycle_id else None`.
    **Does NOT commit** — the UoW owns the commit, so the audit row lands in the SAME txn as the change
    (PLAN §7). Returns the new `event_hash`.
- **DB-side enforcement (UPDATE/DELETE triggers, INSERT/SELECT-only grants) is Platform & Data's M1** — this
  is only the application-layer producer (two layers is a control, one is a convention).
- **What breaks without it:** no tamper-evident audit chain → the system can't prove the buyer's process
  wasn't altered (the entire auditability premise, CLAUDE.md req #2). Census cross-ref: #81.

#### `audit/recorder.py` (census #80, 2141 B) — tenant resolution for decision-site events
- **What:** two resolvers that walk the FK spine to recover the owning tenant (`client_id`) for an event,
  because the governed decision sites hold a `cycle_id`/`award_id`, not a tenant, but the chain keys on
  `client_id` (Gap G-B).
- `client_id_for_cycle(session, cycle_id)` (lines 20-37): `SELECT co.client_id FROM cyc.cycle cy JOIN
  ref.commodity co ON co.id::text = cy.commodity_id WHERE cy.cycle_id=:cid`; raises `ValueError("no client_id
  resolvable for cycle_id=…")` if the row/`client_id` is missing — **the chain cannot key an event without a
  tenant**, so it fails loud rather than guess.
- `client_id_for_award(session, award_id)` (lines 40-57): `SELECT co.client_id FROM awd.award a JOIN cyc.cycle
  cy ON cy.cycle_id = a.cycle_id JOIN ref.commodity co ON co.id::text = cy.commodity_id WHERE a.award_id=:aid`;
  same ValueError discipline.
- **WHY kept separate from `AuditWriter`:** the writer stays a pure chain-appender; the SQL lives in one place;
  literal strings with bound params (S-rule clean). **What breaks without it:** the decision-site services

#### `audit/guards.py` (census #79, 3368 B) — **immutability guard listeners (the app-layer half)**
- **What:** SQLAlchemy `before_update`/`before_delete` mapper listeners that refuse to mutate/delete sealed

- Constants: `SEALED_FLAG = "is_sealed"`, `FROZEN_FLAG = "frozen_at"`.

  principal, **never** from a body/query/header (the #1 cross-tenant-leak vector — security/PLAN §1).
- `from_principal(principal)` → `TenantContext(tenant_id=principal.tenant_id, tenant_code=
  principal.tenant_code)`.
- `GUC_KEY = "app.current_tenant"`; `guc_value` property → `str(self.tenant_id)` — the Postgres GUC the UoW is
  meant to `SET LOCAL` per txn so RLS sees the same tenant (Platform & Data's M10 owns the RLS backstop).
- **GOTCHA / drift:** like the rest of core/security, `TenantContext` is only consumed by `get_tenant` (unused
  by routes) and the domain `ref.service` constructor (out of B5 scope). **No route derives or applies a
  tenant context;** the UoW (`db/session.py`) does NOT `SET LOCAL app.current_tenant` (verified — no GUC set
  in `unit_of_work`). So the application-layer tenant scoping the docstring describes is **not active on the
  request path** in this slice. Census cross-ref: #95.

## 7. Gaps, drift, and unverifiable items (for the caller)

1. **RBAC fully defined, zero enforcement (highest-severity drift).** `require_permission` + the entire
   `ROLE_PERMISSIONS` matrix are attached to NO route. Governed Approver-only transitions (freeze/finalize/
   set-strategy/adjust) are callable by any authenticated `CurrentUser`. Proven by grep (no
   `Depends(require_permission` anywhere in `app/api/**`).
2. **Tenant scoping not applied on the request path.** `TenantContext`/`get_tenant` exist but no route uses
   them; `unit_of_work()` issues no `SET LOCAL app.current_tenant` GUC, so the app-layer tenant filter +
   RLS-GUC handshake described in `tenant.py` is inert in this slice. (The domain repositories may still
   filter by a stamped `client_id` — that is B3/B4 scope, not verifiable here.)
3. **Two parallel auth systems.** `app.core.security.Principal` (set only by the dev stub in `main.py`,
   `None` in prod) vs `app.auth.CurrentUser` (cookie session, what every route actually uses). Audit-event
   `actor` on live routes is `user.username`, not `principal.actor`.
4. **Stale docstring in `router.py:3-6`** still lists `bids` and `runs` among the "present-but-empty"
   routers; both are fully implemented. Cosmetic, but a documentation-vs-code mismatch.
5. **Four empty stub routers (`awards`, `cycles`, `documents`, `ingest`) are dead mounts.** Their designed
   capabilities all already ship on the `runs`/`bids` surface, so they host no routes and 404 on any path.
   Not an MVP cut (capability exists elsewhere) but unused scaffolding — a cleanliness gap. They remain
   load-bearing for `router.py:12`'s single import statement.
6. **`/ready` surfaces a DB outage as a generic 500 `internal_error`**, not a tailored readiness code — by
   design (it relies on the catch-all handler) but worth noting the envelope is `internal_error`, not e.g.
   `not_ready`.
7. **API responses down-cast `Numeric(18,6)` money to JSON `float`** (`bids.py:_num`, `runs.py`
   `AdjustmentLineChange.new_price: float`). Storage precision is preserved; the wire representation is a
   float. The inbound adjustment path re-protects precision via `Decimal(str(new_price))` (`runs.py:914`).
8. **Unverifiable from this slice (correctly deferred):** the actual emission of FROZEN/CLOSED/CREATED audit
   events, the cycle/award FK spine, `PilotService` internals, `app.auth.security` crypto, and `main.py`'s
   middleware — all referenced here but owned by B6/B2/B4. Claims about them rest on the docstrings + the
   call sites I could see, and are flagged as such.
9. **`get_settings()` is `@lru_cache`d** — a process-wide singleton; env changes after first call don't take
   effect without clearing the cache (test gotcha, not a runtime bug).
